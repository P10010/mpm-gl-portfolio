mpm-gl

Interactive fluid simulation driven by sound. An MPM solver with an offline audio analysis pipeline that extracts per-frame features from any song and maps them to simulation and rendering parameters.

01Details

Beat impulse

On each beat, every particle gets a velocity kick in a direction sampled uniformly from the sphere. The direction changes every beat, so over time the kicks cancel out and the cloud does not drift.

Hi-hats

The high-frequency band (contrast-stretched) drives a 3D Perlin noise velocity field sampled at each particle's position. Three domain-shifted scalar noise evaluations give the x, y, z components; the field is re-seeded every frame. The spatial scale is low (0.5), so the perturbation is broad and smooth across the cloud.

Trail decay

The renderer draws into a persistent FBO that is never cleared. A fade pass each frame blends the previous image toward the background color. Spectral flux controls how much of the previous frame survives: more flux means shorter trails, less flux means longer ones.

Fluid–body interaction

Water uses a Tait equation of state; the solid body uses neo-Hookean hyperelasticity. Both materials live on the same MPM grid, so momentum transfer between them happens through the normal P2G/G2P cycle. There is no collision detection or explicit coupling code.

03Technical

Simulation

MLS-MPM with APIC momentum transfer and quadratic B-spline weights. Water uses a Tait equation of state with isotropic deformation gradient reset; the solid body uses neo-Hookean hyperelasticity with a full deformation gradient. Both materials share one grid, so coupling between them is implicit through the P2G / update / G2P cycle. Twenty substeps per frame keep the explicit integrator stable.

C++17 Eigen MLS-MPM Tait EOS neo-Hookean

Rendering

Each particle is an instanced billboarded quad expanded in view space. All particles are drawn into a persistent FBO. A fade pass each frame decays the previous image; additive blending on the particle pass accumulates brightness where particles overlap. Each particle contributes very little light on its own, so the glow you see is hundreds of them stacking. Color is read from a speed-based ramp, one per material.

OpenGL 4.6 GLSL FBO instanced rendering additive blending

Audio analysis

A Python script analyses any audio file and writes one JSON object per frame at 24fps. Each frame contains loudness, beats, spectral change, and energy in four frequency bands (sub-bass through high). Raw features tend to stay near their average, so they are contrast-stretched against a rolling median: most of the time they sit near zero and only spike on actual changes. The C++ side loads the JSON with a hand-written parser (no external dependency).

Python librosa STFT contrast stretching JSON

Audio to visual mapping

Beats inject velocity impulses in a random direction. The high-frequency band drives Perlin noise perturbation across all particles. Spectral flux shortens trail persistence. Loudness scales particle size. Accumulated beat energy triggers macro events: radial explosions from the cloud's center of mass, or uniform pushes in a random direction, with a cooldown to prevent them from firing back to back.

04Extensions I would love to work on

Real-time audio analysis

The audio pipeline currently runs offline: a Python script writes per-frame features to JSON before the simulation starts. Replacing this with a real-time FFT on a live audio input would let the system run from a microphone or line-in, with no pre-processing step.

Additional materials

The MPM framework supports arbitrary constitutive models through the stress computation in P2G. Cloth (thin shells, XPBD membranes) or granular media (sand, snow) would look completely different but use the same grid and renderer.

Interactive parameter tuning

All audio-to-visual weights are currently loaded from a configuration file. An artist-facing interface with sliders and immediate visual feedback would make the system much more usable.
